# AI-Ассистент в Telegram — Технические требования

## Что строим

SaaS-продукт: персональный AI-ассистент в Telegram для конечного пользователя.
Пользователь платит подписку → получает своего Telegram-бота с AI → настраивает его под себя.

**Под капотом:**
- **Kubernetes** — оркестрация сервисов, отдельные деплойменты под процессы с разной нагрузкой и жизненным циклом
- Мультипользовательская архитектура (shared OpenClaw, изоляция по `user_id` / session)
- **LLM (v1):** конечный пользователь подключает **свою** подписку на ИИ — **BYO** (OAuth провайдера и/или сценарий в духе официального OpenClaw: CLI + OAuth, см. документацию провайдера). Расход токенов идёт в **квоту пользователя**, не в единый API-ключ оператора
- Монетизация продукта: **подписка на сервис** (Telegram-ассистент, лимиты, поддержка); оплата за ИИ у провайдера — **на стороне пользователя**
- Всё управление через Telegram: онбординг (в т.ч. привязка ИИ), настройка, биллинг

**Цель:** ноль ручных операций со стороны владельца после запуска.

**Итерации оплаты за ИИ (roadmap):**

| Итерация | Содержание |
|----------|------------|
| **v1 (первая)** | **BYO:** пользователь подключает свою подписку / OAuth (или эквивалент, поддерживаемый OpenClaw для выбранного провайдера). Учётные данные и токены обновления — **хранить зашифрованно** (БД + ключ в Secrets), политика отзыва доступа |
| **v2+** | Опционально: **центральный** API-ключ провайдера (оператор платит за токены); гибридные тарифы; упрощённый онбординг без OAuth |
| **позже** | **Ресейл** подписок на AI через нас (агрегатор биллинга провайдеров) — отдельная продуктовая и юридическая проработка |

## Стек

```
Язык:           Python 3.11+
Telegram:       aiogram 3.x (async)
База данных:    PostgreSQL + SQLAlchemy (async)
Миграции:       Alembic
OpenClaw:       REST API (self-hosted), отдельный Deployment/Service в кластере
LLM (v1):       подключение **пользовательских** учёток (OAuth / сценарии OpenClaw); без единого ключа оператора на всех
LLM (v2+):      опционально центральный Anthropic / OpenAI API и т.д.
Платежи:        ЮKassa (СНГ) или Stripe
Очередь:        Redis + Celery (раздельные воркеры — см. ниже)
Конфиг:         pydantic-settings; секреты кластера — Telegram, платёжка, `OPENCLAW_API_KEY`, **ключ шифрования** полей БД (токены OAuth пользователей)
Деплой:         Kubernetes (манифесты или Helm/Kustomize)
Локальная разработка: Docker Compose (тот же набор сервисов упрощённо)
```

### Очереди и воркеры (разделение по смыслу)

Один процесс `celery worker` на всё допустим на старте, но в проде под Kubernetes логично разнести **хотя бы**:

| Компонент | Назначение | Масштабирование |
|-----------|------------|-----------------|
| **Celery worker — `billing`** | проверка истёкших подписок, синхронизация статусов с платёжкой | реплики по нагрузке |
| **Celery worker — `notifications`** | напоминания об оплате, прочие уведомления в Telegram | реплики по очереди |
| **Celery beat** | расписание (cron-эквивалент) | **ровно 1** реплика (или замена на Kubernetes CronJob) |

При росте нагрузки — отдельные **именованные очереди** в Redis и отдельные Deployments с `celery -Q queue_name`, чтобы тяжёлые задачи не блокировали лёгкие.

## Архитектура в Kubernetes (логика)

```
                    ┌─────────────┐
  Telegram ────────►│  Ingress    │
                    │  (HTTPS)    │
                    └──────┬──────┘
                           │
         ┌─────────────────┼─────────────────┐
         ▼                 ▼                 ▼
   ┌───────────┐   ┌──────────────┐   ┌─────────────┐
   │ bot       │   │ HTTP webhook │   │ OpenClaw    │
   │ Deployment│   │ (billing)    │   │ Deployment  │
   │ (aiogram) │   │ small svc    │   │ REST        │
   └─────┬─────┘   └──────┬───────┘   └──────┬──────┘
         │                │                  │
         └────────────────┼──────────────────┘
                          ▼
              ┌───────────────────────┐
              │ PostgreSQL            │
              │ (managed или в кластере)│
              └───────────┬───────────┘
                          │
              ┌───────────┴───────────┐
              ▼                       ▼
       ┌────────────┐          ┌──────────────┐
       │ Redis      │          │ Celery       │
       │            │          │ workers      │
       │            │          │ + beat       │
       └────────────┘          └──────────────┘
```

*Упрощение на схеме:* при **long polling** Telegram не ходит в ваш кластер; **Ingress** нужен для **вебхука платежей** и — если включён **webhook Telegram** — для приёма апдейтов. Иначе бот сам опрашивает Telegram API.

- **bot** — **long polling** (исходящие запросы к Telegram API, внешний Ingress для самого Telegram не нужен) или **webhook** (тогда Telegram бьёт в ваш HTTPS — нужны Ingress и стабильный URL; несколько подов — с общим состоянием или sticky sessions по политике Telegram).
- **openclaw** — отдельный сервис (внутренний DNS вида `http://openclaw:PORT`), не смешивать процесс с ботом, чтобы масштабировать и обновлять независимо.
- **PostgreSQL / Redis** — для продакшена с перспективой роста предпочтительны **управляемые** (RDS, Cloud SQL, ElastiCache, Memorystore и т.д.): меньше операционных рисков, проще бэкапы и HA.

### Инфраструктурные элементы (по необходимости)

- Ingress-контроллер (nginx, traefik, CNI-поставщик облака)
- TLS (cert-manager + Let’s Encrypt или сертификат облака)
- Логи/метрики: агент на нодах или sidecar, совместимый с вашим стеком наблюдаемости
- HPA для **bot** и **workers** по CPU или по длине очереди Redis (кастомные метрики)

### Изоляция пользователей (один OpenClaw / «Clawbot» на сервере)

Один деплоймент OpenClaw на кластере **нормален**: мультиарендность достигается **логически**, а не отдельным физическим сервером на каждого клиента. Данные разных пользователей **не должны пересекаться**, если соблюдены правила ниже; при ошибке в коде или конфиге OpenClaw возможна утечка контекста — это инженерный риск, а не «магия одного сервера».

**Граница 1 — Telegram-бот (ваш код):**

- Каждый входящий апдейт принадлежит ровно одному `from_user.id`; обработчик **никогда** не подмешивает чужой `user_id`.
- Все запросы к OpenClaw идут только с **`openclaw_session_id` и учётными данными, прочитанными из строки этого пользователя** в PostgreSQL (или из защищённого хранилища, привязанного к `user_id`).
- Ключи Redis для rate limit: префикс с `user_id` (например `ratelimit:{user_id}:{date}`), без общих счётчиков между пользователями.

**Граница 2 — OpenClaw (сессии и память разговора):**

- У каждого пользователя **свой** идентификатор сессии в OpenClaw (`openclaw_session_id` в БД), создаётся при первом обращении и **не переиспользуется** между аккаунтами.
- Вызовы REST API OpenClaw: в теле запроса всегда указывается сессия **текущего** пользователя; контекст диалога и инструменты на стороне OpenClaw привязаны к этой сессии (как именно — по [документации OpenClaw](https://docs.openclaw.ai/) / актуальному API).
- **v1 BYO:** токены OAuth / привязка провайдера хранятся **пер-пользователь** (шифрование в БД); при запросе к LLM используется только цепочка учётных данных этого пользователя.

**Граница 3 — что проверить в конфигурации OpenClaw:**

- Нет ли режима «один общий workspace / одна сессия по умолчанию для всех» без маршрутизации по `session_id` (актуальные ключи конфига — в официальной схеме). После изменений — регрессионные тесты: два тестовых пользователя, параллельные диалоги, убедиться, что ответы не перемешиваются.
- Логи приложения: не логировать полные тексты чужих сообщений; при отладке — маскировка идентификаторов.

**Итог для ответа «пересекутся ли данные»:** при **корректной** реализации слоя бота и **корректной** модели сессий OpenClaw — **нет**, пользователи разделены идентификаторами и сессиями. Один сервер — это общий **процесс**, не общий **контекст чата**. Риск пересечения — как у любого multi-tenant SaaS: баг маршрутизации, общий кэш без ключа `user_id`, ошибка в маппинге `telegram_id → session_id`.

## Модули и логика

### 1. Онбординг (`/start`)

Новый пользователь проходит диалог с кнопками — не команды, а inline-кнопки:

```
Шаг 1: Приветствие + объяснение продукта
        [Начать настройку →]

Шаг 2: "Как назвать твоего ассистента?"
        [Макс] [Алекс] [Мия] [Своё имя]

Шаг 3: "Для чего будешь использовать?"
        [Работа и задачи] [Личный помощник]
        [Изучение языков]  [Своё]

Шаг 4: "Язык общения?"
        [Русский] [English] [Оба]

Шаг 5: Выбор тарифа
        [Базовый $12/мес] [Стандарт $20/мес] [Про $35/мес]
        (описание каждого тарифа)

Шаг 6: Переход к оплате (подписка на **сервис**)
        [Оплатить] → ссылка на платёжную страницу

Шаг 7 (v1): Подключение **своей** подписки на ИИ (BYO)
        Ссылка/инструкция в духе OpenClaw onboard (OAuth или сценарий провайдера).
        До успешной привязки — чат с моделью недоступен или ограничен демо.

Шаг 8 (после оплаты и привязки ИИ): "Готово! Твой ассистент {name} настроен."
        Первое тестовое сообщение ассистента.
```

**Требования к FSM:**
- Каждый шаг — отдельное состояние aiogram FSM
- Пользователь может нажать "Назад" на любом шаге
- Если пользователь бросил онбординг — при следующем `/start` продолжить с того же шага
- **Запись в БД:** после первого осмысленного шага (например `/start` + согласие) допускается **создать или обновить** строку `users` по `telegram_user_id` с `is_onboarded = false`, чтобы хранить прогресс онбординга, состояние FSM, **черновики BYO** (без хранения секретов в открытом виде) и не терять пользователя между сессиями. Полный «боевой» профиль и `is_onboarded = true` — после прохождения обязательных шагов (в v1: оплата сервиса при необходимости + успешная привязка ИИ). Альтернатива — вынести черновик в отдельную таблицу `onboarding_state`, связанную с `users`, если не хотите частично заполнять `users`.

### 2. Основной чат

После онбординга каждое сообщение пользователя:

```python
# Псевдокод flow обработки сообщения
async def handle_message(message):
    user = await get_user(message.from_user.id)

    # 1. Проверка подписки на сервис (middleware)
    if not user.is_active:
        await message.answer("Подписка истекла. [Продлить]")
        return

    # 1b. v1: привязка своей подписки на ИИ (BYO)
    if not user.has_ai_provider_linked:
        await message.answer("Подключите ИИ в настройках / пройдите привязку.")
        return

    # 2. Проверка rate limit (middleware)
    if await is_rate_limited(user.id):
        await message.answer("Достигнут дневной лимит сообщений.")
        return

    # 3. Отправить в OpenClaw с контекстом пользователя
    response = await openclaw_service.send(
        session_id=user.openclaw_session_id,
        message=message.text,
        system_prompt=build_system_prompt(user),  # имя + роль + язык
    )

    # 4. Залогировать токены
    await log_message(user.id, tokens=response.tokens_used)

    # 5. Отправить ответ
    await message.answer(response.text)
```

**Системный промпт строится динамически:**
```python
def build_system_prompt(user: User) -> str:
    return f"""
Тебя зовут {user.assistant_name}.
{user.assistant_role or 'Ты полезный персональный ассистент.'}
Общайся на {user.language}.
{'Запоминай контекст разговора.' if user.memory_enabled else ''}
""".strip()
```

### 6. Rate Limiting

```python
# Лимиты по тарифам
DAILY_LIMITS = {
    'basic':    50,   # сообщений в день
    'standard': 150,
    'pro':      500,
}

# Храним в Redis: key = f"ratelimit:{user_id}:{date}", value = count, TTL = 24h
async def is_rate_limited(user_id: int, plan: str) -> bool:
    count = await redis.get(f"ratelimit:{user_id}:{today()}")
    return int(count or 0) >= DAILY_LIMITS[plan]

async def increment_usage(user_id: int):
    key = f"ratelimit:{user_id}:{today()}"
    await redis.incr(key)
    await redis.expire(key, 86400)
```

## Деплой: Kubernetes

Продакшен ориентируем на **Kubernetes**: отдельные Deployment для `bot`, `openclaw`, воркеров и (при необходимости) для HTTP-сервиса вебхуков; `Service` для внутренней связности; `Ingress` для внешнего HTTPS.

**Минимальный набор сущностей (концептуально):**

- `Deployment` + `Service` — **telegram-bot**
- `Deployment` + `Service` — **openclaw**
- `Deployment` — **celery-worker** (один или несколько с разными очередями)
- `Deployment` — **celery-beat** (1 реплика) *или* `CronJob` вместо beat для части задач
- `Secret` — токен Telegram, платёжка, `OPENCLAW_API_KEY`, **ключ приложения для шифрования** пользовательских OAuth/refresh в БД (не путать с ключами провайдера LLM на всех — в v1 их нет)
- `ConfigMap` — немаскируемые параметры (лимиты, URL managed DB)
- При webhook для Telegram — маршрут на поды бота или отдельный лёгкий HTTP-сервис

**Локально** сохраняем **Docker Compose** (один файл, все сервисы на машине разработчика) — паритет с продом по контейнерам, но без k8s-обвязки.

## Оценка ресурсов сервера (вменяемый ориентир)

Цифры ниже — **не SLA**, а планирование ёмкости; фактическое потребление зависит от OpenClaw, модели, RPS и размера очередей Celery.

### Фаза A — ранний прод / MVP (сотни активных пользователей, умеренный трафик)

| Роль | Рекомендация |
|------|----------------|
| **Ноды кластера** | 2 ноды по **4 vCPU / 8 GB RAM** (или 1 нода **8 vCPU / 16 GB** только если допустим single point of failure на время пилота) |
| **bot** | запросы к LLM идут через OpenClaw; типично **256–512 Mi** лимит памяти на под, **0.25–0.5** CPU |
| **openclaw** | обычно тяжелее бота; закладывать **512 Mi–1 Gi** RAM, **0.5–1** CPU на под до замеров |
| **Celery workers** | старт **256–512 Mi** на воркер, **0.25 CPU**; при росте очередей — добавлять реплики |
| **Celery beat** | **128–256 Mi**, минимальный CPU |
| **PostgreSQL** | если **в кластере**: выделить **1–2 Gi** RAM минимум под небольшую базу; лучше **managed DB** с классом **db.t3.small** или аналог (~2 Gi) |
| **Redis** | **256–512 Mi** для брокера очередей и rate limit; при managed — маленький инстанс |

Итого по «железу под всё»: ориентир **8–16 GB RAM** суммарно на рабочую нагрузку MVP при colocation на нодах, плюс запас на kube-system и пики. Если Postgres и Redis вынести в managed-сервисы, **ноды под приложение** можно начать с **2× (2–4 vCPU, 4–8 GB)**.

### Фаза B — рост (тысячи MAU, выше RPS, длинные очереди)

- Горизонтальное масштабирование: **HPA** на `bot` и `celery-worker`, при необходимости отдельный пул нод для воркеров.
- **OpenClaw**: отдельная группа нод или увеличение реплик, если API становится узким местом (после метрик).
- **PostgreSQL**: увеличение класса инстанса, connection pooling (PgBouncer), индексы и миграции по метрикам.
- **Redis**: при лимите — отдельный инстанс побольше или кластер (редко нужен на ранней стадии).

### Фаза C — заметное масштабирование

- Несколько нод **8+ vCPU / 16–32 GB**, разделение **stateless** (bot, workers) и **stateful** (только то, что остаётся в кластере по политике).
- Региональное размещение, бэкапы БД, disaster recovery — по требованиям бизнеса.

**Практическое правило:** начать с **2 нод по 4 vCPU / 8 GB** + **managed PostgreSQL и Redis**; собирать метрики (CPU, память, latency OpenClaw, длина очереди Redis, ошибки LLM) и пересматривать размеры раз в спринт при росте.

## Что НЕ нужно делать

- Не строить веб-интерфейс (только Telegram)
- Не делать отдельный бот на каждого пользователя (один бот, мультиарендность по user_id)
- Не хранить тексты сообщений пользователей в своей БД (только метаданные и токены)
- Не реализовывать собственную LLM — только API-клиент к OpenClaw
- В **v1** не делать **центральную** оплату токенов оператором и **ресейл** подписок провайдеров — это **следующие итерации** (см. таблицу roadmap выше)

## Порядок разработки (рекомендуемый)

1. `models/` + миграции + `database.py` — фундамент (в т.ч. поля BYO / шифрование)
2. `/start` + онбординг FSM (без оплаты, хардкод активации)
3. **Поток BYO:** привязка учётной записи ИИ пользователя (OAuth / ссылки / согласование с OpenClaw)
4. Основной чат + OpenClaw интеграция (запросы с контекстом **пользовательского** провайдера)
5. Rate limiting + middleware подписки на **сервис**
6. `/settings` меню (в т.ч. «переподключить ИИ» / статус привязки)
7. Биллинг (ЮKassa webhook)
8. Cron-задачи (просроченные подписки, напоминания)
9. Admin-команды
10. Docker Compose (локально) + манифесты Kubernetes + выкат в тестовый кластер

## Критерии готовности

- [ ] Новый пользователь проходит онбординг без единого текстового ввода (только кнопки), кроме шагов, где провайдер требует ввода (OAuth в браузере — вне Telegram)
- [ ] Пользователь может **привязать свою** подписку на ИИ (BYO); статус привязки и ошибки отражаются понятно
- [ ] После оплаты подписка на **сервис** активируется автоматически в течение 30 секунд
- [ ] Истёкшая подписка блокирует чат автоматически
- [ ] Смена настроек ассистента применяется к следующему же сообщению
- [ ] Rate limit работает корректно и сбрасывается в полночь
- [ ] `/reset` сбрасывает историю в OpenClaw
- [ ] Все ошибки логируются, пользователь получает понятное сообщение об ошибке
